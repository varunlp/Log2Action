DEVOPS AND PLATFORM ENGINEERING LEARNING ROADMAP
================================================

Purpose
-------

This roadmap is designed for deep, hands-on DevOps and platform engineering
learning using your existing application as the base project.

The goal is not to rush through tools.
The goal is to understand how production systems actually work.

Current status:
- Application already built
- Containerized with Docker
- Docker Compose used
- Basic Jenkins familiarity
- Linux familiarity
- Product support background
- Strong desire for systems-level understanding

Long-term goal:
Become genuinely strong in DevOps/platform engineering through hands-on,
incremental, production-minded learning.

Core principle:
Every tool should be understood through the system problem it solves.

Always ask:
- What problem does this solve?
- What abstraction does it introduce?
- What does it hide?
- How do I debug below the abstraction?
- What failure modes does this create?
- How would this behave in production?


Learning Path Overview
----------------------

Your project should evolve through these stages:

local process
  -> Docker container
  -> Docker Compose system
  -> CI pipeline
  -> cloud VM
  -> managed cloud infrastructure
  -> Terraform-managed infrastructure
  -> Kubernetes deployment
  -> observable production-like system
  -> GitOps-managed platform

The real deliverable is not only the repository.
The real deliverable is your ability to reason through:

process -> container -> network -> pipeline -> infrastructure -> orchestrator -> observability -> operations


Weekly Operating Model
----------------------

Use this rhythm every week:

Day 1: Concepts
- Study the core ideas.
- Draw diagrams.
- Explain the system in your own words.

Day 2: Build
- Apply the concept manually to your project.
- Avoid excessive automation.
- Prefer simple, inspectable steps.

Day 3: Inspect
- Use command-line tools to observe the system.
- Check processes, ports, logs, files, networks, resources, and state.

Day 4: Break
- Intentionally introduce failure.
- Make the system misbehave in controlled ways.

Day 5: Debug
- Diagnose using evidence.
- Avoid guessing.
- Form hypotheses and test them.

Day 6: Document
- Write notes, runbooks, diagrams, and lessons learned.

Day 7: Review
- Can I explain this without looking it up?
- Can I debug this when it fails?
- Do I understand what this tool hides?
- Can I operate this under pressure?


WEEK 1: LINUX PROCESSES AND RUNTIME FOUNDATIONS
===============================================

Concepts
--------

Learn how Linux sees your application before Docker or Kubernetes exist.

Focus areas:
- Processes
- PIDs
- Parent and child processes
- Environment variables
- File descriptors
- Signals
- Exit codes
- stdout and stderr
- CPU, memory, disk, and network basics

Why this matters:
Every production system eventually becomes processes running on machines.
Docker and Kubernetes do not remove this reality. They wrap and manage it.

Practical Tasks
---------------

Run your app directly on Linux without Docker.

Inspect the process:

ps aux
pstree
top
htop
lsof -p <pid>
cat /proc/<pid>/status
cat /proc/<pid>/environ

Start the app, then stop it with different signals:

kill -TERM <pid>
kill -KILL <pid>

Observe:
- Does the app shut down cleanly?
- Does it flush logs?
- Does it close connections?
- Does it leave child processes behind?

Debugging Exercises
-------------------

Break the app deliberately:
- Start it with a missing environment variable.
- Bind it to a port already in use.
- Kill a dependency while it is running.
- Fill the disk or temp directory.
- Run it with insufficient permissions.

Observability
-------------

Capture:
- CPU usage
- Memory usage
- Open files
- Listening ports
- Logs
- Exit code

Useful commands:

ss -lntp
journalctl
dmesg
df -h
free -m
top
lsof

Reflection Points
-----------------

Answer these in your notes:
- What exactly is "the application" from the OS point of view?
- What does it mean for a process to be healthy?
- What happens when the parent shell exits?
- What does Docker later abstract away?
- What is the difference between a crash and a clean exit?

Production Engineering Thinking
-------------------------------

A production service is a process with discipline around:
- startup
- shutdown
- configuration
- logging
- recovery
- resource limits
- permissions


WEEK 2: LINUX NETWORKING FROM FIRST PRINCIPLES
==============================================

Concepts
--------

Focus areas:
- IP addresses
- ports
- TCP and UDP
- loopback
- network interfaces
- DNS resolution
- routing
- NAT
- firewalls
- connection refused vs timeout

Why this matters:
Many production outages are network misunderstandings, not application bugs.

Practical Tasks
---------------

Run your app locally and inspect how clients reach it.

Commands:

ip addr
ip route
ss -tulpen
curl -v http://localhost:<port>
dig example.com
nslookup example.com
traceroute example.com

If available:

tcpdump -i lo port <port>

Debugging Exercises
-------------------

Create these failures:
- App listens on 127.0.0.1 but you try to access it externally.
- App listens on the wrong port.
- DNS lookup fails.
- Firewall blocks traffic.
- A dependency hostname resolves but the port is closed.

Use curl to distinguish:
- DNS failure
- TCP connection failure
- HTTP response failure
- application-level error

Observability
-------------

Create a network debugging checklist:

1. Is DNS resolving?
2. Is the route valid?
3. Is the port listening?
4. Is traffic allowed by firewall/security group?
5. Is the app responding?
6. Is the response correct?

Reflection Points
-----------------

Answer:
- What is the difference between connection refused and timeout?
- What does localhost mean?
- Why does localhost change meaning inside containers?
- What does a listening socket prove?
- What does it not prove?

Production Engineering Thinking
-------------------------------

Production network issues often involve:
- DNS
- routing
- firewalls
- security groups
- load balancers
- dependency timeouts
- wrong bind addresses


WEEK 3: DOCKER IMAGE INTERNALS
==============================

Concepts
--------

Focus areas:
- Images vs containers
- Dockerfile instructions
- Layers
- Build context
- Cache behavior
- Tags
- Digests
- Base images
- Image size

Why this matters:
A Docker image is a deployable artifact. If you do not understand how it is
built, you cannot reason about reproducibility, security, or runtime behavior.

Practical Tasks
---------------

Inspect your image:

docker image ls
docker history <image>
docker inspect <image>
docker build --no-cache -t app:test .

Experiment:
- Reorder Dockerfile instructions.
- Move frequently changing files lower in the Dockerfile.
- Add and remove files from the build context.
- Add a .dockerignore file if needed.
- Compare build time and image size.

Debugging Exercises
-------------------

Break the build intentionally:
- Reference a file outside the build context.
- Use a wrong working directory.
- Break dependency installation.
- Use a missing base image tag.
- Add large unnecessary files to the image.

Observability
-------------

Track:
- image size
- number of layers
- rebuild time
- cache hits and misses
- base image used

Commands:

docker history <image>
docker inspect <image>
du -sh .

Reflection Points
-----------------

Answer:
- What gets sent to the Docker daemon during build?
- Why does Docker cache some layers but not others?
- Why are small images operationally useful?
- What are the risks of unknown or outdated base images?
- What is the difference between tag and digest?

Production Engineering Thinking
-------------------------------

A production image should be:
- reproducible
- minimal
- secure
- pinned where appropriate
- easy to scan
- fast to pull
- deterministic to build


WEEK 4: DOCKER CONTAINERS AND ISOLATION
=======================================

Concepts
--------

Focus areas:
- Container lifecycle
- Namespaces
- cgroups
- Filesystem isolation
- Process isolation
- PID 1 behavior
- Signals
- Resource limits

Why this matters:
Containers are isolated Linux processes, not full virtual machines.

Practical Tasks
---------------

Run your container interactively:

docker run --rm -it <image> sh
docker ps
docker inspect <container>
docker exec -it <container> sh

Inspect from host and container:

ps aux
lsns
mount
hostname
cat /proc/1/status

Run with resource limits:

docker run --memory=256m --cpus=0.5 <image>

Debugging Exercises
-------------------

Create failures:
- Container exits immediately.
- App does not stop gracefully.
- Container cannot write to a path.
- Memory limit causes crash.
- Wrong CMD or ENTRYPOINT.
- Logs are written to a file instead of stdout/stderr.

Observability
-------------

Commands:

docker stats
docker logs <container>
docker top <container>
docker inspect <container>

Observe:
- process tree
- memory usage
- CPU usage
- restart behavior
- exit code

Reflection Points
-----------------

Answer:
- Is a container a lightweight VM? Why or why not?
- What is shared with the host?
- Why does PID 1 matter?
- How do cgroups enforce resource limits?
- What happens to container filesystem changes after removal?

Production Engineering Thinking
-------------------------------

Containers fail in production because of:
- bad startup assumptions
- missing config
- poor signal handling
- resource limits
- filesystem assumptions
- poor logging behavior

Observability

Reflection Points
-----------------

- Why does localhost inside a container refer to itself?
- What does Docker DNS provide?

Production Engineering Thinking
-------------------------------

Production systems must treat networking and state explicitly.
Containers are disposable. Data usually is not.

WEEK 9: AWS FOUNDATIONS WITHOUT ABSTRACTION

Study Rules
===========

1. Do not optimize for speed.
2. Do not blindly copy commands.
3. Before running a command, know what question it answers.
4. After running a command, inspect the output carefully.
5. Break things intentionally.
6. Debug from evidence.
7. Keep notes.
8. Prefer understanding over memorization.
9. Learn what each abstraction hides.
10. Always connect local behavior to production behavior.


First Session Starting Point
============================

Start with Week 1.

Your first task:
Run the application directly on Linux without Docker.

Then inspect:

ps aux
ss -lntp
lsof -p <pid>
cat /proc/<pid>/status
curl -v http://localhost:<port>

Write down:
- process ID
- listening port
- memory usage
- environment variables required
- how the app behaves when stopped with SIGTERM
- what logs are produced

Do not move to Docker internals until you can explain your application as a
Linux process.
